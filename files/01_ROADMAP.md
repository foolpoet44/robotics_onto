# 01 · 로드맵 (P0 → P4)

진단(`00_DIAGNOSIS.md`)을 실행 순서로 옮긴 것이다. 가치/위험/의존 관계를 고려해 5단계로
나눴다. **위에서 아래로** 진행한다. 각 단계는 그 자체로 빌드·타입체크·검증 게이트를 통과한
상태로 끝나야 한다(언제 멈춰도 저장소가 초록불).

## 시퀀싱 원칙

- **저위험·고정리(housekeeping) 먼저** → 토대를 단단히 하고 표면적을 줄인 뒤 큰 수술을 한다.
- **렌더링(P1)을 데이터 실질화(P2)보다 먼저** → 서버 컴포넌트로 전환해 두면, 관계망이
  생긴 뒤 그것을 빌드 타임에 굽기 쉽다.
- **검증 강화(P3)는 데이터 실질화(P2) 직후** → 새로 채운 관계망/URI를 게이트로 잠근다.
- **시각화 등 표면(P4)은 맨 마지막** → 연결할 엣지(P2)가 생긴 다음에야 의미가 있다.

```
P0 토대 정합 ─┬─→ P1 렌더링 정상화 ─┐
              │                      ├─→ P2 온톨로지 실질화 ─→ P3 검증 게이트 강화 ─→ P4 표면
              └──────────────────────┘
```

---

## P0 · 토대 정합 (Foundation Coherence) — 반나절~1일 · 위험 낮음

진단 F-4, F-6 대응. 큰 수술 전에 복제·드리프트·위생을 정리한다.

- **P0-1** `RobotSkill` 중복 제거 — `app/page.tsx`의 인라인 인터페이스를 삭제하고
  `app/lib/robotics-data.ts`에서 import. 인라인 fetch/도메인 목록/도메인명 맵을 lib 함수로 교체.
- **P0-2** 조직 카드 카운트 파생화 — `organizations/page.tsx`의 하드코딩 숫자를 데이터에서 계산.
- **P0-3** 공유 UI 상태 컴포넌트 — `LoadingState` / `EmptyState` / `ErrorState`를
  `app/components/`에 만들고 모든 페이지가 동일하게 사용. "조용한 실패" 제거.
- **P0-4** `tsconfig` `target` → `es2020`. 빌드·타입체크 재확인.

**P0 종료 조건**: 페이지에 타입/도메인/fetch 복제가 없다. 모든 데이터 화면이 로딩·빈·에러
상태를 명시적으로 표현한다. 게이트 3종 초록.

---

## P1 · 렌더링 아키텍처 정상화 (Server-First) — 1~2일 · 위험 중간

진단 F-3 대응. 클라이언트 SPA를 Next 15 서버 컴포넌트 + 정적 생성으로 되돌린다.

- **P1-1** 빌드 타임 데이터 접근 — `fetch("/data/...")`를 **빌드 시 파일 읽기/직접 import**로
  교체(서버 컴포넌트에서 `import data from ...` 또는 `fs` 읽기). 데이터 로더를 lib에 일원화.
- **P1-2** `generateStaticParams` 추가 — `domains/[domain]`, `organizations/[orgId]`를 정적
  파라미터로 사전 생성. 빌드 표에서 `ƒ Dynamic` → `●/○ Static`이 되는지 확인.
- **P1-3** 서버/클라이언트 경계 분리(아일랜드) — 데이터·콘텐츠는 서버 컴포넌트가 렌더,
  검색·필터 같은 상호작용만 작은 `"use client"` 아일랜드로 분리. 초기 HTML에 콘텐츠가 담기게.

**P1 종료 조건**: `next build` 라우트 표에서 동적 셸이 사라지고 정적 프리렌더로 바뀐다.
첫 HTML에 스킬/도메인 콘텐츠가 포함된다(로딩 깜빡임 제거). 게이트 3종 초록.

---

## P2 · 온톨로지 실질화 (Make It an Ontology) — 2~4일 · 위험 중간 · ★핵심

진단 F-1, F-2 대응. 이름값(`_onto`)을 실제로 채운다. 설계는 `03_DATA_ONTOLOGY_SPEC.md` 참조.

- **P2-1** 관계 의미론 정의 — `related_skills`를 단순 ID 배열에서, 관계 유형을 가진 엣지로
  격상(prerequisite / co-required / specialization 등). 스펙 문서의 모델 확정.
- **P2-2** 관계 생성 로직 — Python 생성기가 도메인·숙련도·역할·부모 트리를 근거로 엣지를
  산출하도록 구현. 최소 커버리지 목표를 둔다(예: 스킬당 평균 ≥ 2 관계, 고립 노드 0).
- **P2-3** ESCO 출처 정직화 — 둘 중 하나를 택해 구현:
  (a) 실제 ESCO API/덤프로 매핑하고 매핑 신뢰도 등급을 부여, 또는
  (b) 내부 식별자임을 드러내는 정직한 URN(`urn:rsf:skill:...`)으로 바꾸고 `esco_uri`는
  매핑이 실재할 때만 채운다(없으면 `null`).
- **P2-4** 화면 반영 — 도메인/스킬 상세에 "관련 스킬" 섹션을 노출(P1의 서버 렌더 위에).

**P2 종료 조건**: `related_skills` 엣지 > 0, 고립 노드 0, dangling 참조 0. ESCO URI는
실재 매핑이거나 정직한 내부 네임스페이스다. 게이트 3종 + 신규 검증(P3 일부) 초록.

---

## P3 · 검증 게이트 강화 (Earned Green) — 1일 · 위험 낮음

진단 F-5 대응. 검증기가 형태가 아니라 의미를 보게 한다.

- **P3-1** 관계 실질 검사 — `related_skills` 커버리지 임계값(비어 있으면 _경고가 아니라 실패_),
  고립 노드 0, 양방향 대칭성(A→B면 B→A 권장) 점검.
- **P3-2** URI 진위/네임스페이스 검사 — `esco_uri`가 ESCO 네임스페이스를 쓰면서 가짜
  슬러그면 실패. 내부 URN 규칙 준수 여부 검사.
- **P3-3** 사이클·일관성 검사 — `parent_skill_id` 순환 탐지, 역할/숙련도 enum 일관성.
- **P3-4** 게이트 연결 — `validate:data`를 pre-commit 훅 또는 CI 스텝으로 묶고, README에 명시.

**P3 종료 조건**: 빈 관계망/가짜 URI가 들어오면 검증이 **빨간불**을 낸다(거짓 안심 종료).

---

## P4 · 표면 다듬기 (Polish) — 0.5~2일 · 위험 낮음 · 선택

진단 F-6 + 확장. P2로 엣지가 생긴 뒤에야 의미 있는 것들.

- **P4-1** 접근성 — 검색 인풋 `<label>`, 전역 skip-link, 포커스 스타일, 색 대비 점검.
- **P4-2** 스타일 일관성 결정 — globals.css vs styled-jsx 중 하나로 통일(권장: 토큰화된
  globals + 페이지별 모듈). 디자인 토큰 정리.
- **P4-3** README 데이터 단일 출처 — 스킬 수 등 수치를 데이터에서 생성하거나 "검증기 출력이
  진실"임을 명시.
- **P4-4** (선택) 관계 시각화 — 엣지가 생겼으니 네트워크 그래프를 *클라이언트 아일랜드*로
  재도입(ESCON의 교훈: SSR 안전하게 `dynamic(..., { ssr:false })` + ErrorBoundary).

**P4 종료 조건**: 키보드만으로 전 페이지 탐색 가능, 스타일 단일 체계, README 수치가 진실.
