# 기술 전문가 육성 프레임워크 검증 맵

## 검증 원칙

육성 프레임워크는 사람의 상태를 다루므로 화면이 예쁘게 보이는 것만으로는 충분하지 않다.
데이터 참조 무결성, 단계 이동 일관성, 개인정보 경계를 각각 검증한다.

| 요구사항                     | 자동 검증                 | 브라우저 검증           | 완료 기준                            |
| ---------------------------- | ------------------------- | ----------------------- | ------------------------------------ |
| 6단계 퍼널 정의              | `validate:development`    | `/development-tracks`   | 진입부터 인증까지 순서대로 표시      |
| 트랙별 목표 인원 설정        | `validate:development`    | 퍼널 카드               | 각 단계 목표값 표시                  |
| 전문 영역과 기존 도메인 연결 | `test:development`        | 전문 영역 분포          | 존재하지 않는 도메인 참조 0          |
| 핵심 스킬 연결               | `test:development`        | 후보자 상세 스킬 링크   | dangling 스킬 참조 0                 |
| 후보자 현재 단계             | `test:development`        | 후보자 목록, 상세       | 목록과 상세의 현재 단계가 일치       |

## 음성 테스트 목록

`scripts/test-development-validation.js`는 최소 아래 실패 시나리오를 포함한다.

1. 존재하지 않는 도메인을 전문 영역에 연결
2. 존재하지 않는 스킬을 핵심 스킬로 연결
3. 단계 `order` 중복
4. 뒤 단계의 목표 인원이 앞 단계보다 증가
5. 후보자의 현재 단계가 트랙에 없음
6. 후보자가 선택한 전문 영역이 트랙에 없음
7. 직능 역량 점수가 허용 범위를 초과
8. 임팩트 과제가 있는데 필수 설명이 비어 있음
9. 후보자의 완료 이력이 현재 단계보다 뒤에 위치

## 브라우저 점검 시나리오

### 퍼널 화면

1. `/development-tracks`에 접속한다.
2. 잠재 후보군부터 Lv3 Tech Leader 인증까지 순서가 맞는지 확인한다.
3. 전문 영역 필터를 바꾸고 후보자 목록이 좁혀지는지 확인한다.
4. 특정 후보자를 눌러 상세 페이지로 이동한다.

### 후보자 상세

1. 현재 단계와 과거 완료 단계가 구분되는지 확인한다.
2. 핵심 스킬 링크를 눌러 `/skills/[skillId]`로 이동한다.
3. 현재 수준과 목표 수준 차이를 읽을 수 있는지 확인한다.
4. 5개 직능 역량과 현장 임팩트 과제 상태가 표시되는지 확인한다.

### 회귀 점검

1. `/organizations/robot-solution`에서 연관 트랙 링크를 확인한다.
2. `/skills/RSF-AMR-001`에서 전문가 중요도 평가 컴포넌트가 유지되는지 확인한다.
3. `/reviews`에서 기존 검수 큐가 유지되는지 확인한다.

## 개인정보 체크리스트

- MVP 후보자는 익명 샘플 ID만 사용한다.
- 실제 사번, 이메일, 조직 인사정보는 정적 JSON에 넣지 않는다.
- 실제 운영 전에는 인증, 접근 권한, 감사 로그, 보존 기간을 결정한다.
- 브라우저 `localStorage`는 개인 실험용으로만 취급하고 팀 공용 기록으로 간주하지 않는다.
