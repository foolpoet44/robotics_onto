# 평가 전용 도메인 페이지 직접 확인 가이드

이 문서는 `/evaluation` 페이지 변경사항을 로컬에서 직접 확인하는 절차를 정리합니다.
평가 페이지는 4대 도메인(메인)과 기능 도메인(서브)의 계층형 중요도 평가,
하위 스킬 조회, 도메인 변경요청을 제공합니다.

## 1. 개발 서버로 화면 확인

```bash
npm install
EVAL_SESSION_SECRET=test npm run dev
```

브라우저에서 다음 주소로 접속합니다.

```text
http://localhost:3000/evaluation
```

확인할 항목은 다음과 같습니다.

- 상단 제목이 `도메인 분류 평가 페이지`로 표시되는지 확인합니다.
- **메인 섹션**: 4대 도메인 카드 4장이 표시되고, 각 카드에 역할·스킬 수·
  구성(기능 도메인 × 스킬 수)·직접 평가 평균·서브 롤업(참고)이 보이는지
  확인합니다.
- **서브 섹션**(세부 기준): 기능 도메인 카드 7장이 표시되고, 각 카드에
  스킬 수·평균 중요도·하위 스킬 조회 토글이 있는지 확인합니다.
- 비로그인 상태에서는 평가 저장 폼 대신 "평가자 로그인 후 가능" 안내가
  표시되는지 확인합니다(조회는 자유).
- 평가자 로그인(예: EVAL-001 / `robot01`) 후 메인·서브 각각 1~5점 평가를
  저장하면 평균·최근 평가 목록이 갱신되고, 평가자명이 자동 적용되는지
  확인합니다(수기 입력 없음).
- 서브 평가를 저장하면 해당 기능 도메인이 속한 4대 도메인 카드의
  "서브 롤업" 값이 스킬 수 가중 평균으로 갱신되는지 확인합니다. 직접
  평가와 롤업의 차이가 1.0점 이상이면 근거 확인 배지가 표시됩니다.
- 하위 스킬 조회에서 스킬별 "도메인 변경요청"(기능/4대 도메인 축, 사유
  필수)이 접수되는지 확인합니다.
- 스킬 상세 목록, 조직 정보, 개인 역량 정보, 육성 트랙, 검수 큐가 평가
  카드 안에 표시되지 않는지 확인합니다.

## 2. 프로덕션 빌드로 라우트 생성 확인

```bash
npm run build
```

빌드 출력의 `Route (app)` 목록에 다음 라우트가 포함되어야 합니다.

```text
/evaluation            (ƒ Dynamic — 세션·서버 평가 데이터 사용)
/api/domain-ratings
/api/domain-change-requests
```

## 3. 타입 검사 확인

```bash
npm run type-check
```

## 4. 내비게이션 진입 확인

개발 서버 실행 후 브라우저에서 아무 페이지에 접속한 뒤 상단 내비게이션의 `평가` 메뉴를 클릭합니다.
클릭 후 주소가 `/evaluation`으로 이동하고 평가 전용 페이지가 표시되는지 확인합니다.

## 5. 평가 목적 적합성 체크리스트

| 검증 항목 | 기대 결과 |
| --- | --- |
| 계층형 평가 | 4대 도메인(메인) 카드 4장 + 기능 도메인(서브) 카드 7장 |
| 부가 정보 제외 | 스킬 상세, 조직, 역량, 육성, 검수 데이터 미표시 |
| 신원 자동 적용 | 로그인 세션에서 평가자명 자동 적용, 수기 입력 없음 |
| 서버 아카이빙 | 평가가 DB(`domain_importance_ratings`) 또는 파일 폴백에 저장 |
| 롤업 정합성 | 서브 평가의 스킬 수 가중 평균이 메인 카드 참고치로 표시, Δ≥1.0 배지 |

적합성 점검 결과와 후속 개선 항목은 `docs/PROJECT_IMPROVEMENTS.md`에서 관리합니다.
